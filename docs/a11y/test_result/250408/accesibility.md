# アクセシビリティ要件と考慮事項

## アクセシビリティとは

アクセシビリティとは、障害のある人も含めて、すべての人がWebサイトやアプリケーションを利用できるようにすることです。これには、視覚障害、聴覚障害、運動障害、認知障害など、様々な障害を持つ人々のニーズを考慮する必要があります。

## 主要なアクセシビリティガイドライン

- **WCAG 2.1 (Web Content Accessibility Guidelines)**: Webコンテンツのアクセシビリティに関する国際的なガイドライン

  - レベルA（最低限必要な要件）
  - レベルAA（一般的に達成すべき要件）
  - レベルAAA（理想的な要件）

- **JIS X 8341-3**: 日本における情報通信アクセシビリティの標準規格

## 主要なアクセシビリティ要件

### 1. 知覚可能性

- **代替テキスト**: 画像、ボタン、アイコンなど非テキストコンテンツには適切な代替テキストを提供する
- **マルチメディアの代替手段**: 動画には字幕や音声解説を提供する
- **適応可能なコンテンツ**: 様々な方法で表示・操作できるようにコンテンツを構造化する
- **識別可能性**: テキストと背景のコントラスト比を適切に確保する

### 2. 操作可能性

- **キーボード操作**: すべての機能をキーボードだけで操作できるようにする
- **十分な時間**: ユーザーがコンテンツを読み、操作するのに十分な時間を提供する
- **発作防止**: 発作を引き起こす可能性のあるコンテンツを避ける
- **ナビゲーション**: ユーザーがナビゲートしやすく、自分の現在位置を確認できるようにする

### 3. 理解可能性

- **読みやすさ**: テキストコンテンツを読みやすく理解しやすくする
- **予測可能性**: ユーザーが予測できる方法でWebページを表示・操作する
- **入力支援**: ユーザーが間違いを回避し、修正できるようにする

### 4. 堅牢性

- **互換性**: 支援技術を含む様々なユーザーエージェントと互換性を持たせる
- **標準準拠**: HTML、CSS、JavaScriptなどの標準技術を適切に使用する

## コンポーネントのアクセシビリティチェックポイント

### ボタン・リンク

- ボタンには明確なラベルを提供する
- 状態（無効、選択中など）が視覚的に識別でき、スクリーンリーダーにも伝わる
- クリック可能な領域が十分に確保されている

### フォーム要素

- すべての入力フィールドに関連するラベルがある
- エラーメッセージが明確で、修正方法が示されている
- 必須フィールドが視覚的に識別でき、スクリーンリーダーにも伝わる

### 画像・アイコン

- すべての意味のある画像に代替テキストがある
- 純粋に装飾的な画像は代替テキストを空にするか、CSSで背景画像として実装する

### モーダル・ダイアログ

- 適切なARIAロールとプロパティが設定されている
- 開いたときにフォーカスが移動し、閉じたときに元の位置に戻る
- Escキーで閉じることができる
- モーダル内にフォーカスがトラップされる

### ドロップダウン・メニュー

- キーボードで開閉・操作できる
- 展開状態がスクリーンリーダーに伝わる
- メニュー項目が適切にグループ化されている

### テーブル

- 適切な見出しセルと関連するデータセルが設定されている
- 複雑なテーブルには要約や説明が提供されている

## テスト方法

1. **自動テスト**: Lighthouse、axe、Wave などのツールを使用
2. **キーボードテスト**: マウスを使わずにすべての機能を操作できるか確認
3. **スクリーンリーダーテスト**: NVDA、VoiceOver、JAWS などのスクリーンリーダーで操作
4. **コントラストチェック**: テキストと背景のコントラスト比が適切か確認
5. **画面拡大テスト**: 200%以上に拡大しても操作可能か確認

## 当社の目標

- すべてのUIコンポーネントはWCAG 2.1 レベルAAに準拠することを目指す
- 利用者のフィードバックを積極的に取り入れ、継続的に改善する
- 設計・開発段階からアクセシビリティを考慮する
